| Browser -> `src/proxy.ts` / Next.js server | Every request crosses Clerk's session-cookie check before reaching any route |
| Browser -> Clerk Frontend API | The hosted `<SignIn/>` widget talks directly to Clerk's own API, not this app's server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-1-01 | Elevation of Privilege | `src/lib/auth/requireStaffAccess.ts` and every future protected Server Component/Action | mitigate | `requireStaffAccess()` is the sole gating function (D-08, FOUND-04); Task 2's grep-based acceptance criterion structurally enforces no other file inlines a gating `auth()` check; `src/proxy.ts` alone is documented as a non-authoritative fast-reject per 01-RESEARCH.md Pitfall 1, never treated as sufficient by itself |
| T-1-02 | Information Disclosure | `src/lib/env.ts`, `.env.local`, Vercel Project Settings env vars | mitigate | `CLERK_SECRET_KEY`/`DATABASE_URL` are never prefixed `NEXT_PUBLIC_`; only `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` is client-exposed; `src/lib/env.ts`'s zod schema names each var explicitly so a future accidental client-prefix is a visible schema change, not a silent leak |
| T-1-SC | Tampering | npm installs (`create-next-app` scaffold + Phase 1 package set) | accept | All 13 Phase 1 packages already passed `slopcheck` with `[OK]` verdicts in 01-RESEARCH.md's Package Legitimacy Audit (2026-07-23); no `[ASSUMED]`/`[SUS]` packages in this plan, so no additional blocking checkpoint is required — re-run `slopcheck` if any installed version diverges from the audited pins above |
