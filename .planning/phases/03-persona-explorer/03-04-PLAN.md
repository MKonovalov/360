Close the single blocking gap from 03-VERIFICATION.md (score 13/15, status `gaps_found`): the Persona list's "Has signals: No" filter option is a silent no-op. It is wired correctly through the UI and URL (`persona-filters.tsx` already emits `?hasSignals=false` correctly — do not touch that file), but the value is destroyed at two points downstream: (1) `parsePersonaFilters` in both `src/app/personas/page.tsx` and `src/app/personas/[id]/page.tsx` collapses `firstValue(params.hasSignals) === 'true'` into a plain boolean, so "no param" and "explicitly false" both become `false`; (2) `listPersonas` in `src/lib/db/queries/personas.ts` only implements the truthy branch (`filters.hasSignals ? exists(...) : undefined`), so `false` and `undefined` behave identically — no filtering happens either way. This also closes the phase's last requirement gap (DATA-01's "browsable, trustworthy" contract per VERIFICATION.md's Requirements Coverage table) so Phase 3 can be re-verified as fully passing.
