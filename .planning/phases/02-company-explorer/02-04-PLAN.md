<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser URL query string → `CompanyFilters` → `listCompanies` WHERE clause | The primary new untrusted-input surface this plan introduces — every filter value now originates from a user-editable URL, not just an internal call site |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-2-01 | Tampering | `searchParams` → `CompanyFilters` → `listCompanies(filters)` | mitigate | `company-filters.tsx`'s enum-typed Selects (`parseAsStringEnum(schemaEnum.enumValues)`) are the client-side validation boundary; server-side, `listCompanies` (Plan 02-01) only ever uses Drizzle's parameterized `eq`/`ilike`/`exists` — a tampered `?signal=drop-table` value fails `parseAsStringEnum` client-side and, even if it reached the server directly via a hand-crafted URL, would simply fail to `eq()`-match any row rather than execute as SQL, since no raw `sql.raw()` interpolation exists anywhere in the query path. |
| T-2-08 | Tampering | Direct (non-UI) URL manipulation bypassing the client-side `parseAsStringEnum` validation entirely (e.g. `curl '/companies?signal=%27; DROP TABLE company;--'`) | mitigate | Server-side defense-in-depth: `listCompanies`'s Drizzle operators are parameterized regardless of what string arrives in `searchParams` — this was already the design in Plan 02-01, Task 2 of this plan does not introduce any new raw-SQL path, only new callers of the existing safe function. |
| T-2-SC | Tampering | `npm install nuqs` | mitigate | Already vetted `[OK]` in `.planning/phases/02-company-explorer/02-RESEARCH.md`'s Package Legitimacy Audit (official docs confirmed this session) — no additional blocking checkpoint required per the Package Legitimacy Gate protocol, since the audit table already covers this exact package. |
</threat_model>
